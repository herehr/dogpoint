Gift Recipients Feature

Implementation Plan · DogPoint Adopce

Overview

Allow subscribers to attach one or more “non-donors” (gift recipients) to their adoption. The gift recipient can see all posts of the adopted animal, but nothing else (no payments, subscriptions, etc.).

Current Architecture

  • Access: GET /api/adoption/me returns access: { [animalId]: true } for animals where the user has an ACTIVE subscription.
  • Posts: Loaded from /api/posts/public; the UI only displays them when hasAccess(animalId) is true.
  • Notifications: notifyUsersAboutNewPost notifies users with ACTIVE subscriptions for that animal.

What Needs to Be Done

1. Database

  • New model: SubscriptionGiftRecipient (or GiftRecipient):
    • subscriptionId → Subscription
    • email (required, for invite)
    • userId (optional, set when they register)
    • displayName (optional)
    • createdAt
  • Migration to add the table and indexes.

2. Backend

Task Description
Extend getMe Include in access all animalIds where the user is a gift recipient of an ACTIVE subscription (match by userId or email).
Extend notifyNewPost Include gift recipients in the notification list (in-app + email).
Gift recipient API CRUD for gift recipients per subscription: list, add (email), remove. Only the subscription owner can manage them.
Link on registration When a user registers with an invited email, link userId in SubscriptionGiftRecipient.

3. Frontend

Task Description
User dashboard Section “Darovat adopci” / “Obdarovaní” per adopted animal: add/remove gift recipients by email.
Access No change needed if backend returns the correct access map.
Invite flow Invite email with link to register (or magic link).

4. Edge Cases

  • Limit gift recipients per subscription (e.g. 5).
  • Only the subscriber can add/remove recipients.
  • If the subscription is canceled, gift recipients lose access.
  • Gift recipient must have a User account (register/login) to see posts.

Hours Estimate

Phase Hours
Database schema + migration 0.5 – 1
Backend: extend getMe + notifyNewPost 1 – 1.5
Backend: gift recipient CRUD API 1.5 – 2
Backend: link userId on registration 0.5 – 1
Frontend: gift recipients UI in user dashboard 2 – 3
Invite email template + sending 0.5 – 1
Testing, edge cases, Czech copy 1 – 2
Total (core feature) ~8 – 11 hours

Optional Extras

  • Magic-link invite (no password for first visit): +2–3 h
  • Admin view of gift recipients: +1 h
  • More robust validation and error handling: +1 h

Summary

Core: New table, backend access + notifications + CRUD API, frontend UI, invite email.

Result: Subscriber adds gift recipients by email; recipients register and see posts for the adopted animal only.

Effort: ~8–11 hours for the core feature.
